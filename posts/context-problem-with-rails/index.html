# 缺乏上下文切换能力的 Rails

Rails 是一个 Ruby 的 Web framework。其核心思想就是 MVC 和 RestFul，然后对资源进行 CRUD。

简单来说，Rails 让你『在知道自己在做什么的时候』可以达到非常高的效率。不过『效率高』和『以正确的方式做事』可不能划等号。

Web 框架大多大同小异，无非由一下几点组成。

  1. 路由，起到分发导航的功能，根据 HTTP 所请求的 URL 和 ACTION （GET POST PUT DELET）等调用对应的 endpoint。
  2. 控制器，就是上面提到的 endpoint，负责接收 HTTP 请求的参数。然后返回 HTTP 的 Response。
  3. 模型，这一层应该是最复杂的，这和应用的具体业务有关，不像其他组件有着固定的固定的处理方式。
  4. 视图，User Interface。

Rails 的一个重大问题就是把整个这一些列组件作为一个整体，虽然组件可以被单独使用，但是不凑在一起就无法发挥威力。（恐龙战队）

回过头想想，那个著名的 15 分钟创建一个博客应用的例子，就是利用了脚手架的自动生成代码，和约定大于配置的规则，其实就是俗称的『把一切问题转化成钉子』的思路。

可是，不是所有问题都能转化为钉子的，就算勉强转了，也是强扭的瓜不甜。

在开发好一个 Rails 应用后，如果去做一些『运营』的工作的话，你可能会发现，很难使用简单的几句代码去实现一个业务逻辑。

就是因为，在 Rails 中，『用例』都是被打撒了的。很难把业务逻辑集中在一个地方。

就拿 『当前用户』来说吧，一般都是在一个 base_controller 中，根据 session 中的数据，取出当前用户的对象。

然后 controller 需要 『知道』这次请求要取出的是『当前用户』的动态。

View 层应该是职责很清楚的，只要把它所知道的数据展示出来就可以了，可是 Rails 中的 View 层真心不给力。

首先是没有地方做简单的逻辑处理，然后 Helper 又都是全局的 Module ！！！完全没有任何 oo 的概念。

至于 ActiveRecord，它的功能是强大的，可是被用错了地方。

ActiveRecord 这个名字出自于 《企业架构实战》中的一个模式，意为：

很明显这个模式的用途是操作数据库（表）的，并不符合我们通常意义上的 （domain） model。

它顶多就是数据库表的一个模型（model）。

既然在 Rails 中，一个 model （比如 User）对应的是一个表（类）和表中的一条记录（实例）。

那么一个 model 的所有方法都是关于数据表的，所谓的 callback，就类似于数据库的触发器。

那这样的架构，和『远古』时代的 sql 脚本的开发方式又有什么区别？

领域模型根本没有生存的空间。比如 post 和 post-content，就是由于表结构的问题，不得不分开定义。

也就是说，在开发 Rails 应用的时候，你既需要知道 HTTP 是怎么一回事，也要知道数据库是怎么一回事。

难怪 Rails 被认为是『不适合初学者的框架』了。

很明显这样什么都要知道的做法是不符合『单一职责』原则的。

我们开发的应用应该只是将 web 作为一种输入和输出的媒介，不应该和具体的业务逻辑有过多的耦合。
