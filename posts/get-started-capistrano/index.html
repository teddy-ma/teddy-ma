capistrano 入门

在我学习 capistrano 的时候, 就觉得相关的资料非常少. 而且这个部署工具很有可能已经由你的前辈们配置好了, 部署时输入命令就行了. 这样就更导致了不少同学根本不会从头配置 capistrano 的情况. 所以我在这里总结一下以供参考.

什么是 capistrano

capistrano 是一个部署工具, 虽然是用 ruby 写成的, 但是它并不限于部署 rails 应用, 相反很多国外资料中提到用它部署 php 的不在少数.

关于版本

capistrano 3 是目前的最新版, 相对于2, 3是一个重写的版本, 本文不会涉及到2的内容, 作为一个单独的部署工具, 没有理由不去升级它.

关于安装

capistrano 是一个 ruby gem， 安装非常简单:

$ gem install capistrano

然后在要部署的项目的根目录下执行：

$ bundle exec cap install

就会生成如下的文件结构（注意： capistrano完全是一个client端的工具， 所以在rails项目中应该 把它安装在 development 的 group 下）

├── Capfile
├── config
│   ├── deploy
│   │   ├── production.rb
│   │   └── staging.rb
│   └── deploy.rb
└── lib
    └── capistrano
            └── tasks

capistrano 的功能

capistrano 3 是基于 rake 的, 通过 ssh 到远程服务器来执行一些列部署的命令. 因为你很难描述 capistrano 到底是干什么的. 因为 理论上只要能连接到服务器并且有权限, 你可以做任何事.

capistrano 的文件结构

先看一下 capistrano 在服务器上留下的文件结构：

  deploy@VM-204-84-ubuntu:~$ tree apps/ -L 2 -A
  apps/
  ├── current -> /home/deploy/apps/releases/20150209075227
  ├── releases
  │   ├── 20150205044932
  │   ├── 20150205052719
  │   ├── 20150205070603
  │   ├── 20150205072850
  │   └── 20150209075227
  ├── repo
  │   ├── branches
  │   ├── config
  │   ├── description
  │   ├── FETCH_HEAD
  │   ├── HEAD
  │   ├── hooks
  │   ├── info
  │   ├── objects
  │   ├── packed-refs
  │   └── refs
  ├── revisions.log
  └── shared
      ├── bin
      ├── bundle
      ├── config
      ├── log
      ├── public
      └── tmp

可以看到， 每次部署后， release 文件夹下会有一个时间戳格式的文件夹， 代表着一次部署 而 current 这个软链接就是当表当前运行着的版本， 是不是和 git 很像。在配合nginx之类的 web服务器一起使用时， nginx上配的路径就是current文件夹的路径。

capistrano 的基本工作原理

最最“原生”的 capistrano 的工作就是在目标服务器上拉取git仓库中的项目，然后发布一个release， 再把current软链接到当前的release版本（这些操作是和项目语言无关的）。当然为了完成一次部署 光这样是远远不够的， 所以需要安装一些插件或者自己写一些命令。

capistrano 的配套插件

以 rails 为例， 部署少不了的操作就是执行数据库的 migrate， 重启 web 服务器等。 这些任务绝大部分都有现成的插件了， 用bundler安装后在 Capfile 中 require 就行了。

capistrano 的基本配置

capistrano 的配置基本就在 config/deploy.rb 和 deploy/ 文件夹下的配置文件这两个地方（lib/capistrano/tasks/） 文件夹下可以写自己的rake task， 不过用的比较少。

deploy.rb 的配置

这里是一些全局的配置，可以设定一些变量， 如 ruby 版本啊， git 仓库什么的。

config/deploy 下的配置

这里的rb文件一般以环境名称命名， 比如 production.rb, staging.rb 等，里面定义了不同环境下的 特殊变量， 如部署文件位置， 环境变量， 分支等。

capistrano 的职责

打一个比方. 服务器就像是一个拎包入住的酒店. capistrano 所发布的应用就是酒店的客人. 有些东西是酒店自带的, 比如床, 马桶, 桌子椅子之类的, 这对应服务器上的 web 服务器, 数据库服务等 而客人就是来来往往的发布版本. 客人会去调用一些基础设施, 比如重启 web 服务器. 而有些设备处于灰色地带, 比如拖鞋, 这个客人可以用酒店的, 也可以用自带的. 我们在使用 capistrano 的时候也要注意, 通常经常变化的部分就是 capistrano 的职责范围. 不太变化的部分交给其他工具来完成.
