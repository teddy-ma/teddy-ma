Backbone 中 el 的意义

今天由于对 backbone 的 el 理解不够透彻，被小小的坑了一把。

先说一下碰到的问题：

一个两步走的功能，在第一个页面点击“下一步”后出现第二个页面。

OK 这很简单，根据给出的静态 html， 网页的主要内容都在一个叫 #main-container 的 div 中。

于是我把 view1 和 view2 的 el 都设置为 #main-container 。

两个 view 的 render 写法一样， 都是：

  $(@.el).html(@.template(model: @.model))

很快就发现问题了，就是在后退后绑定的 click 事件被执行了两次。这情况我见过，很明显是view没有被回收。

于是就先在 Router 中就清除了当前存在的 view 。这样一来我发现原本写在静态 html 中的 div#main-container 都不见了！

原来 view 被 remove 掉的时候会连带着自己的 el 元素一起被清空的（即使这个 el 元素是原本就存在于页面上的）！

于是我果断改变策略， 把 render 的逻辑改成：

  $("#main-container").html(@template(model: @model))

这样页面是正常了，但是很快就发现：view 中定义的事件都不起作用了！

于是我去查看了一下 backbone 文档（也许是我查的不仔细），只是隐约看到了一些线索：

Omitting the selector causes the event to be bound to the view's root element (this.el)

总之意思就是绑定的事件元素都是要在 view 的 el 中的。那么很明显了，render 的逻辑需要写成：

  tmp = @.template(model: @model)
  $(@el).html(tmp)
  $("#main-container").html(@el)

这个 el 是一个空的 div， 总之这下 template 中的 dom 都处于这个 el 之内了， 那事件绑定也就能正常工作了！
